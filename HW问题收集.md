## HW 问题收集

列举在HW 1、HW1.1过程里，你所遇到的2\~3个通过自己学习已经解决的问题，和2\~3个尚未解决的问题与挑战

### 已解决

1. 什么是领域驱动设计（DDD-lite）
   1. **上下文**：在最初 Homework 1 中，我写出了 Sudoku 和 Game 类。但在Homework 2接入 Svelte 时，由于完全不清楚架构分层，我不知道该如何继续进行来让写的类接入UI。
   2. **解决手段**：通过网络搜索与向 AI提问探讨，我理解了 DDD-lite 的架构设计与分层理念。我明确了 View 层与 Domain 层之间的界限要求，并认识到了两者之间需要一个 Application 层来作为缓冲。
2. 什么是 Svelte 的编译时机制与 Svelte Store？如何跨越“响应式边界”？
   1. **上下文**：在尝试将 Game 的纯逻辑接入 UI 时，我遇到了如何确保界面随着数据改变而动态刷新的潜在问题。直接调用 game.guess() 修改内部二维数组无法触发 Svelte 的更新。
   2. **解决手段**：通过网络搜索与向 AI提问探讨 Svelte 3 的相关工作原理，包括Svelte Store、响应式边界，订阅机制与回调函数、"$"符号及语法糖等内容，我认识到 Svelte 是一个依赖静态分析和 = 赋值操作的编译器。我学习了 Svelte Store 的 subscribe 订阅契约。最终按照作业要求推荐的采用了 Store Adapter 模式，将领域对象封装在闭包中，通过对外暴露的方法执行业务逻辑后，统一调用底层 store 的 set来实现领域数据变化驱动 UI 自动刷新。

### 未解决

1. 如何彻底将关键业务状态收敛到领域层，实现“单一事实来源（Single Source of Truth）”

   1. **上下文**：在此次 Codex Review 的“缺点 1”中指出，我目前的架构里Sudoku 和 Game 虽然负责了落子操作，但invalidCells 和 gameWon 等衍生状态依然是 Svelte Store 层基于获取到的二维数组，利用已有的 Store 层代码重新计算的。这不仅使得一份底层数据具有多个拷贝，破坏了单一事实原则，更破坏了 OOD 的封闭性，导致实现规则被拆散在 Domain 和 Store 。

   2. **尝试解决手段**：我仍在考虑如何将 checkWin() 和 getConflictingCells() 方法移入 Sudoku 内部。是每次 set 时都调用这些方法生成一个快照抛给 UI，还是让 UI 暴露在局部的 Derived Store 中？一些问题还有待我与codex确认。

2. 领域对象的严格规则防御与 UI 的“容错交互语义”之间该如何做到对齐？

   1. **上下文**：Codex Review中缺点 2 和 3 指出我的代码存在业务语义不一致。目前我的 Sudoku.guess 出于领域层设计，会直接拒绝任何形成冲突的非法输入。但实际上原版游戏 UI 的交互语义是“允许输入，但将其标红高亮”。由于我的 API 直接丢弃了非法输入且未返回详尽的命令结果，导致当前实现中 UI 完全失去了给出错误反馈的能力。
   2. **尝试解决手段**：这可能需要重构领域层的命令API，比如让 Game.guess 从返回一个 Boolean改写成返回一个类似 { status: 'SUCCESS' | 'CONFLICT', conflictingCells: [...] } 的结果对象，以此包含输入正确性的信息。我需要在保护数据一致性前提下让接口提供足够的上下文。
